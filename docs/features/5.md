# Threat Overlay and ELK Auto-Layout (DFD-05)

**What it does:** Adds threat-state visualisation (colour-coded badges on DFD components and data flows) and automatic left-to-right graph layout using ELK.js, so you can see threat coverage at a glance and position new diagrams without manual placement.

## Threat Overlay

### How it works

You call `applyThreatOverlay(graph, otm, risks?)` after loading an OTM document into a graph. The function reads the `threats[]` array on each OTM component and data flow and applies colour-coded state to the matching JointJS cells.

Each component receives an `overlayThreats` attribute with `{ count, state }` — the count of threats and the worst (highest-severity) state across all threats on that component. You use `badgeColourForState(state)` to get the fill, stroke, and text colours for a badge at that state.

Data flow links are coloured directly: **red** (`#D32F2F`) when any threat on the flow is `exposed` or `identified`, **grey** (`#616161`) otherwise.

Trust zone containers get a red border when any embedded child component has an `exposed` or `identified` threat.

An optional `risks[]` argument supplements the OTM threats. Each entry is matched to a component or data flow by `moduleId`. This lets external risk records (from a separate risk register) contribute to the overlay without modifying the OTM document.

### Badge colour mapping

| Colour | Threat states |
|--------|--------------|
| Red | `exposed`, `identified` |
| Yellow | `assessed`, `planned`, `implementing` |
| Green | `closed`, `eliminated` |
| Grey | `accepted`, `monitored`, `deferred`, `out-of-scope`, `transferred`, `escalated`, `expired` |

When a component carries threats in multiple states, the worst state determines the badge colour (red takes priority over yellow, yellow over green, green over grey).

### Clearing the overlay

Call `clearThreatOverlay(graph)` to remove all overlay state. This resets `overlayThreats` to `null` on every cell, restores link stroke colours to grey, and restores trust zone borders to grey. Call this before re-applying the overlay after a data refresh.

### Badge click events

`ThreatBadge` emits `badge:pointerclick` on the host cell view when the user clicks a badge. The event payload includes the cell's `id`. Listen on the paper:

```javascript
paper.on('cell:badge:pointerclick', (cellView, id, evt) => {
    console.log('Clicked badge on cell', id);
});
```

### Example

```javascript
import { applyThreatOverlay, clearThreatOverlay, badgeColourForState, ThreatBadge } from '@joint/dfd';

// Apply overlay after loading the graph from OTM.
applyThreatOverlay(graph, otmDocument);

// Render badges using the overlayThreats attribute.
graph.getElements().forEach(element => {
    const overlay = element.get('overlayThreats');
    if (!overlay) return;
    const view = paper.findViewByModel(element);
    ThreatBadge.attach(view, { count: overlay.count, state: overlay.state });
});

// Clear before refreshing data.
clearThreatOverlay(graph);
```

## ELK Auto-Layout

### How it works

You call `elkLayout(graph, options?)` to position all elements using the ELK.js `layered` algorithm. The function converts the JointJS graph to an ELK model, runs the layout, then writes computed positions back to each element.

`elkLayout` is **async** and returns a `Promise<graph>`. Await it before rendering badges or reading positions.

The layout direction is **left-to-right** — data sources appear on the left, consumers on the right. This mirrors the natural reading direction of DFD notation.

Trust zone containers become ELK compound nodes. Components embedded in a zone become children of that compound node. After layout, ELK resizes the container to enclose all its children with padding.

### When to call

- On first render when the OTM document has no position data in `representations[]`. `otmToGraph` detects this and fires `elkLayout` automatically (fire-and-forget). Await `elkLayout(graph)` directly if you need guaranteed positions before rendering.
- When the user clicks an "Auto Layout" button.
- Do **not** call automatically when the OTM already carries positions — that would discard manual placement.

### Options

| Option | Default | Description |
|--------|---------|-------------|
| `direction` | `'RIGHT'` | ELK layout direction. `'RIGHT'` is left-to-right. |
| `nodeSpacing` | `30` | Minimum spacing between sibling nodes (pixels). |
| `layerSpacing` | `60` | Spacing between layers (columns) in the layout (pixels). |
| `padding` | `40` | Padding inside trust zone containers around their children (pixels). |

### Example

```javascript
import { elkLayout } from '@joint/dfd';

// After otmToGraph when no positions exist:
await elkLayout(graph);

// With custom spacing:
await elkLayout(graph, { nodeSpacing: 50, layerSpacing: 80, padding: 60 });
```

### Error handling

If ELK fails (for example, due to a cyclic graph or an unsupported configuration), `elkLayout` silently returns the graph with elements at their initial positions. It does not throw.

## Configuration

`elkjs` is a direct dependency of `@joint/dfd` (version pinned to `0.11.1`). It is not bundled — consumers get it from the package's `dependencies` and can tree-shake unused ELK algorithms.

## Known limitations

- `applyThreatOverlay` stores overlay state as a model attribute, not as an actual `ThreatBadge` view. Callers are responsible for reading `overlayThreats` and attaching `ThreatBadge` highlighters in the paper's `cell:added` or update handlers.
- `elkLayout` positions are absolute coordinates. If you resize the paper or scroll, the diagram layout remains valid but may need a `paper.fitToContent()` call to bring all elements into view.
- Cross-zone data flows (edges between nodes in different compound nodes) are placed at the root level of the ELK graph. ELK resolves the endpoints automatically, but edge routing may cross container boundaries visually.
